# feature-flags Specification

## Purpose
Type-safe runtime feature flag infrastructure for gating enterprise auth, voice, changelog, and automations cutover. Backed by Drizzle feature_flag_overrides table.
## Requirements
### Requirement: Typed default values source of truth

The system SHALL define all feature flag keys and their default values in a single TypeScript const map (`FLAG_DEFAULTS`) located at `src/main/lib/feature-flags.ts`. The map SHALL be declared with `as const` so that TypeScript can infer the exact value type for each key.

The map MUST include at minimum these four entries on the initial ship of the feature:

- `enterpriseAuthEnabled: false`
- `voiceViaLiteLLM: false`
- `changelogSelfHosted: false`
- `automationsSelfHosted: false`

#### Scenario: Adding a new flag

- **WHEN** a developer needs to introduce a new feature flag
- **THEN** they add one entry to `FLAG_DEFAULTS` with a literal default value
- **AND** the TypeScript type of `getFlag("the-new-flag")` is automatically inferred as the literal type of that default
- **AND** no schema migration is needed (the flag is keyed off the hardcoded map)

#### Scenario: Retrieving a flag that exists in defaults but has no override

- **WHEN** application code calls `getFlag("enterpriseAuthEnabled")` with no row in the `feature_flag_overrides` table
- **THEN** the function returns the default value from `FLAG_DEFAULTS` (`false` in this case)
- **AND** the return type is statically inferred as `boolean` (not `unknown` or `any`)

### Requirement: Persistent override table

The system SHALL provide a Drizzle-backed SQLite table `feature_flag_overrides` with the following columns:

- `key` — `text`, primary key, matches a key from `FLAG_DEFAULTS`
- `value` — `text`, NOT NULL, JSON-encoded (string, number, boolean, or plain object)
- `updatedAt` — `integer` (timestamp mode), NOT NULL, defaults to now at insert

The table SHALL persist in the same SQLite database as all other application data (`{userData}/data/agents.db`), and SHALL be created via a Drizzle migration generated by `bun run db:generate`.

#### Scenario: Setting an override for the first time

- **WHEN** application code calls `setFlag("enterpriseAuthEnabled", true)`
- **THEN** the system inserts a row into `feature_flag_overrides` with `key = "enterpriseAuthEnabled"`, `value = "true"`, and `updatedAt = now()`
- **AND** subsequent calls to `getFlag("enterpriseAuthEnabled")` return `true` until the override is cleared

#### Scenario: Updating an existing override

- **WHEN** application code calls `setFlag("enterpriseAuthEnabled", true)` and the key already has an override row
- **THEN** the system UPDATEs the existing row rather than inserting a duplicate
- **AND** `updatedAt` is refreshed to the current timestamp

#### Scenario: Clearing an override

- **WHEN** application code calls `clearFlag("enterpriseAuthEnabled")`
- **THEN** the system DELETEs the override row for that key
- **AND** subsequent calls to `getFlag("enterpriseAuthEnabled")` return the default value from `FLAG_DEFAULTS`

### Requirement: Type-safe public API

The system SHALL expose four functions from `src/main/lib/feature-flags.ts`:

- `getFlag<K extends keyof typeof FLAG_DEFAULTS>(key: K): (typeof FLAG_DEFAULTS)[K]`
- `setFlag<K extends keyof typeof FLAG_DEFAULTS>(key: K, value: (typeof FLAG_DEFAULTS)[K]): void`
- `clearFlag<K extends keyof typeof FLAG_DEFAULTS>(key: K): void`
- `getAllFlagsWithSources(): Array<{ key: string; value: unknown; source: "default" | "override"; updatedAt: Date | null }>`

The first three MUST reject (at compile time) any key not present in `FLAG_DEFAULTS`.

#### Scenario: Unknown flag key rejected at compile time

- **WHEN** application code calls `getFlag("flagThatDoesNotExist")`
- **THEN** TypeScript emits a compile error because the key is not a member of `keyof typeof FLAG_DEFAULTS`
- **AND** the call site cannot be built until either the flag is added to `FLAG_DEFAULTS` or the call is corrected

#### Scenario: Value type mismatch rejected at compile time

- **WHEN** application code calls `setFlag("enterpriseAuthEnabled", "yes")` (string when the flag is boolean)
- **THEN** TypeScript emits a compile error because `"yes"` is not assignable to `boolean`
- **AND** the error points at the value argument, not the key

### Requirement: JSON serialization boundary

The system SHALL serialize override values as JSON strings before writing to the database, and parse JSON strings when reading. Any value that is not a valid JSON-stringifiable primitive or plain object MUST cause `setFlag` to throw before any database write occurs.

#### Scenario: Valid primitive serialized

- **WHEN** application code calls `setFlag("enterpriseAuthEnabled", true)`
- **THEN** the `value` column stores the literal string `"true"`
- **AND** `getFlag("enterpriseAuthEnabled")` round-trips to boolean `true`

#### Scenario: Function value rejected

- **WHEN** application code attempts `setFlag("someFlag", () => 42)` (assuming `someFlag` had a function default)
- **THEN** `setFlag` throws before touching the database
- **AND** no override row is created

### Requirement: tRPC router exposure

The system SHALL expose a `featureFlags` tRPC router at `src/main/lib/trpc/routers/feature-flags.ts` with four procedures:

- `list` — query returning `getAllFlagsWithSources()`
- `get` — query accepting `{ key: string }` and returning the effective value
- `set` — mutation accepting `{ key: string; value: unknown }` and calling `setFlag`
- `clear` — mutation accepting `{ key: string }` and calling `clearFlag`

The router MUST be mounted in `createAppRouter` at `src/main/lib/trpc/routers/index.ts`.

#### Scenario: Router mounted in app router

- **WHEN** the application starts and `createAppRouter` runs
- **THEN** `featureFlags` appears as a key on the returned router object
- **AND** a tRPC client in the renderer can call `trpc.featureFlags.list.query()` and get back the current flag state

### Requirement: Server-side plan resolution from feature flags
The system SHALL provide a server-side function `resolveUserPlan(userId: string): { email: string, plan: string, status: string }` in the `1code-api` service that maps the user's feature flag state to a subscription plan identifier. In enterprise mode, all authenticated users SHALL receive `plan: "onecode_max"` with `status: "active"`. The plan identifiers SHALL match the upstream SaaS values expected by the desktop app (`onecode_pro`, `onecode_max_100`, `onecode_max`).

#### Scenario: Enterprise user plan resolution
- **WHEN** the `1code-api` service receives `GET /api/desktop/user/plan` from an authenticated enterprise user
- **THEN** the service returns `{ "email": "<from-gateway-header>", "plan": "onecode_max", "status": "active" }` without consulting any external billing system

#### Scenario: Plan identifier matches desktop app expectations
- **WHEN** the desktop app's `auth-manager.ts:434` fetches `GET /api/desktop/user/plan`
- **THEN** the `plan` field contains one of: `onecode_pro`, `onecode_max_100`, `onecode_max`
- **AND** the `voice.ts:90-115` subscription check passes for enterprise users (gated on `onecode_pro` or higher)

### Requirement: Auxiliary-AI feature flags

The `FLAG_DEFAULTS` export in `src/main/lib/feature-flags.ts` SHALL include four new entries governing auxiliary-AI features (chat title generation, commit message generation):

- `auxAiEnabled: true` — boolean master kill switch. When false, both `generateChatTitle` and `generateCommitMessage` return the truncated fallback without any provider call.
- `auxAiModel: ""` — string override for the Anthropic-mode model id. Empty string means "use the precedence chain: `mode.modelMap.haiku` when available, else `claude-3-5-haiku-latest`". Non-empty string forces this exact model id regardless of the active account's configuration.
- `auxAiTimeoutMs: 5000` — number. Per-call timeout for provider requests, used to construct the `AbortSignal` passed to the Anthropic SDK.
- `auxAiOrigin: ""` — string. Reserved for future operator tuning (forced LiteLLM endpoint override). Empty string means "use the active account's endpoint". The current implementation does not consume this flag; it is reserved to avoid a future breaking flag-rename.

All four flags SHALL be persistable via the existing `feature_flag_overrides` table and readable via `getFlag<K>(key)`. No env var counterparts are introduced — the flags are the sole configuration surface for auxiliary-AI behavior.

#### Scenario: Default values support out-of-the-box auxiliary-AI

- **WHEN** no `feature_flag_overrides` rows exist for the `auxAi*` keys
- **THEN** `getFlag("auxAiEnabled")` returns `true`
- **AND** `getFlag("auxAiModel")` returns `""` (empty string — triggers the resolution precedence chain)
- **AND** `getFlag("auxAiTimeoutMs")` returns `5000`
- **AND** `getFlag("auxAiOrigin")` returns `""`

#### Scenario: Kill switch disables provider calls

- **WHEN** `setFlag("auxAiEnabled", false)` has been called
- **AND** a chat is created
- **THEN** `generateChatTitle` and `generateCommitMessage` both return `getFallbackName(...)` immediately
- **AND** no Anthropic SDK or Ollama call is attempted

#### Scenario: Model override survives restart

- **WHEN** `setFlag("auxAiModel", "claude-sonnet-4-5")` has been called
- **AND** the app restarts
- **THEN** the override persists via `feature_flag_overrides` table
- **AND** `getFlag("auxAiModel")` returns `"claude-sonnet-4-5"` after restart

#### Scenario: Type safety — auxAiEnabled is inferred as boolean

- **WHEN** TypeScript code calls `getFlag("auxAiEnabled")`
- **THEN** the return value is inferred as `boolean` (not `boolean | string | number`)
- **AND** TypeScript code calls `getFlag("auxAiModel")` returns `string`
- **AND** TypeScript code calls `getFlag("auxAiTimeoutMs")` returns `number`
