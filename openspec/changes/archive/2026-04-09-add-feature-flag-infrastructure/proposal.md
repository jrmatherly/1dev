## Why

The enterprise auth migration in `.scratchpad/auth-strategy-envoy-gateway.md` v2.1 cannot land as a flag day — the new authentication path (Entra ID bearer tokens injected via `ANTHROPIC_AUTH_TOKEN` through an Envoy Gateway) must coexist with the existing `CLAUDE_CODE_OAUTH_TOKEN` path during a Strangler Fig rollout. Without a feature-flag mechanism, every migration gate requires a hard cutover branch, and every rollback means reverting code instead of flipping a switch.

The 1Code codebase has **no existing feature flag infrastructure**. The closest analog is `anthropicSettings` (a singleton row with `activeAccountId` at `src/main/lib/db/schema/index.ts:125`) which is purpose-built for multi-account state, not generic boolean/string/number configuration. Every Phase 0, Phase 1, and Phase 2 gate in strategy v2.1 §6 depends on being able to gate new code paths behind runtime-toggleable flags.

This change introduces the minimum infrastructure to unblock that migration. It is a greenfield addition — no data migration, no backwards-compat shims.

## What Changes

- **ADD** new Drizzle table `feature_flag_overrides` (key/value/updatedAt) that stores user/operator overrides of default flag values
- **ADD** `src/main/lib/feature-flags.ts` module exposing:
  - `FLAG_DEFAULTS` — const map of flag keys to default values (source of truth for flag types)
  - `getFlag<K>(key)` — returns the effective value (override if present, default otherwise) with full type inference
  - `setFlag<K>(key, value)` — writes an override row
  - `clearFlag<K>(key)` — removes the override, reverting to default
  - `getAllFlagsWithSources()` — debug/admin helper showing which flags are overridden vs. default
- **ADD** new tRPC router `featureFlags` at `src/main/lib/trpc/routers/feature-flags.ts` with `list`, `get`, `set`, and `clear` procedures (for eventual admin UI use)
- **ADD** the router to `createAppRouter` composition at `src/main/lib/trpc/routers/index.ts` (21 routers total, up from 20)
- **ADD** Drizzle migration generated via `bun run db:generate`
- **ADD** regression guard test at `tests/regression/feature-flags-shape.test.ts` asserting the table exists in the migration directory and that `FLAG_DEFAULTS` contains the flags that strategy v2.1 depends on
- **SEED** initial flag set in `FLAG_DEFAULTS`:
  - `enterpriseAuthEnabled: false` — gates the new Entra/Envoy code path in `claude.ts` (Gate #8 dependent)
  - `voiceViaLiteLLM: false` — gates the F4 voice transcription cutover
  - `changelogSelfHosted: false` — gates the F6 changelog endpoint cutover
  - `automationsSelfHosted: false` — gates the F2 automations backend cutover

No existing functionality changes. All existing call sites continue to work unchanged; the only use of the new infrastructure in this change is the regression test.

## Capabilities

### New Capabilities

- `feature-flags`: Runtime-toggleable configuration values with typed defaults and persistent overrides. The capability covers the storage layer (Drizzle table), the TypeScript API (`getFlag`/`setFlag`/`clearFlag`/`getAllFlagsWithSources`), the tRPC exposure for eventual admin UI, and the invariants around type safety and default-vs-override semantics.

### Modified Capabilities

None. This is greenfield — no existing specs touch feature flag concerns.

## Impact

**Affected code:**
- New file: `src/main/lib/feature-flags.ts` (~150 LOC)
- New file: `src/main/lib/trpc/routers/feature-flags.ts` (~40 LOC)
- Modified: `src/main/lib/db/schema/index.ts` (+10 LOC — one new `sqliteTable` declaration + type exports)
- Modified: `src/main/lib/trpc/routers/index.ts` (+2 LOC — import and mount)
- New migration: `drizzle/NNNN_add_feature_flag_overrides.sql` (generated by `bun run db:generate`)
- New test: `tests/regression/feature-flags-shape.test.ts` (~60 LOC)

**Affected APIs:** None (no existing tRPC procedures change). One new router added.

**Dependencies:** None. Uses existing `drizzle-orm`, `better-sqlite3`, `@trpc/server` — all already in `package.json`.

**Breaking changes:** None. This change is purely additive; removing any of the added code restores the pre-change state.

**Downstream blockers unblocked:**
- Phase 0 hard gate #8 (`add-anthropic-auth-token-injection`) — needs `enterpriseAuthEnabled` flag to gate the new env-var injection
- Phase 0 hard gate #12 (this change — unblocks itself as the "infrastructure" entry)
- Phase 1 MSAL integration (`add-msal-token-acquisition`) — needs `enterpriseAuthEnabled` to gate MSAL token acquisition
- All F2/F4/F6 cutover changes in Phase 2 — each needs its own flag
