# Database rules — Drizzle schema + SQLite

The database layer uses **Drizzle ORM** against a **SQLite** database stored at `{userData}/data/agents.db`.

## Source of truth

- **Schema:** `src/main/lib/db/schema/index.ts` — the `sqliteTable(...)` calls are authoritative
- **Migrations:** `drizzle/*.sql` + `drizzle/meta/*_snapshot.json` (generated by `drizzle-kit`)
- **Canonical doc:** `docs/architecture/database.md`

## Rules

1. **Never edit `drizzle/*.sql` files by hand.** Run `bun run db:generate` to regenerate after a schema change.
2. **Never call `bun run db:push` in production paths.** It's dev-only and can corrupt migration state.
3. **Never edit `drizzle/meta/_journal.json` by hand.** It must stay in lockstep with the `.sql` files.
4. **Auto-migration runs on startup** via `initDatabase()` in `src/main/lib/db/index.ts`. If ground truth has new tables but no corresponding migration, the app will fail to start in dev mode.
5. **After adding or removing a table**, run the `db-schema-auditor` agent to detect drift in documentation surfaces (CLAUDE.md, PROJECT_INDEX.md, `docs/architecture/database.md`).

## Query patterns

```typescript
import { getDatabase, projects, chats } from "../lib/db"
import { eq } from "drizzle-orm"

const db = getDatabase()
const allProjects = db.select().from(projects).all()
const projectChats = db.select().from(chats).where(eq(chats.projectId, id)).all()
```

## Migration workflow

1. Edit `src/main/lib/db/schema/index.ts`
2. Run `bun run db:generate` — creates a new `drizzle/NNNN_*.sql` + snapshot
3. Start the app (`bun run dev`) — `initDatabase()` applies the migration
4. Run the `db-schema-auditor` agent to verify all documentation surfaces match
5. Update `docs/architecture/database.md` with the new table/column
